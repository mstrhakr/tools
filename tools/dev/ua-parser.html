    How User-Agent strings work & why they’re a mess

    A User-Agent (UA) string is sent by the browser in every HTTP request to tell the server what client it is. They were meant to be simple, but decades of browser history made them absurd — a modern Chrome UA looks something like this:

    Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/122.0.0.0 Safari/537.36

    That string says Mozilla, WebKit, like Gecko, Safari — and it's actually Chrome. Here's why:

    • Netscape (Mozilla) was dominant, so sites served good content only to Mozilla/ UA strings — other browsers started spoofing it to get the full site.
    • IE added like Gecko to pass Gecko-detection scripts. Safari inherited WebKit tokens. Chrome inherited both WebKit and Safari tokens.
    • Every browser now claims to be every other browser for compatibility reasons.

    Parsing is done client-side using regex pattern matching against known browser and OS signatures — the same approach used by libraries like ua-parser-js. User-Agent parsing is inherently heuristic; sufficiently unusual or spoofed strings may produce incorrect results.
